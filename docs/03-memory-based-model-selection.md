# 메모리 기준으로 Gemma 4 모델 고르는 방법

[메인 안내로 돌아가기](../gemma4-local-setup-guide.md)

## 이 문서에서 다루는 내용

- 공식 모델 스펙과 로컬 배포 크기 기준 메모리 판단법
- 노트북/데스크톱 메모리로 어떤 모델을 고르면 되는지
- 준비용 간단 계산식
- 8GB, 16GB, 32GB, 36GB 환경별 예시

## 먼저 알아둘 점

- 아래 표는 **공식 모델 스펙**과 **LM Studio / Ollama에 표시되는 로컬 배포 크기**를 함께 보는 기준입니다.
- 표시 크기는 다운로드 파일 또는 로컬 배포 패키지 기준이라, 실제 실행 중 최대 메모리 사용량과 항상 같지는 않습니다.
- 운영체제, 실행 도구, 브라우저, 긴 대화를 빠르게 이어가기 위해 저장하는 내부 메모리(KV cache), 긴 컨텍스트, 멀티모달 입력으로 인한 추가 메모리는 별도로 필요합니다.
- 따라서 실제 개인 노트북에서는 **표의 숫자보다 여유를 더 잡아야** 합니다.
- Apple Silicon Mac처럼 **unified memory**를 쓰는 환경에서는 전체 메모리를 하나의 풀처럼 보면 됩니다.
- Windows/Linux의 **전용 GPU 노트북**은 시스템 메모리(RAM)뿐 아니라 **GPU 메모리(VRAM)** 도 같이 확인해 주세요.

## 로컬 배포 크기와 컨텍스트 참고

2026-04-27 기준 LM Studio 모델 페이지와 Ollama Gemma 4 태그 페이지에 표시되는 기본 항목입니다.

| 모델 | LM Studio 표시 크기 | Ollama 기본 태그 표시 크기 | 컨텍스트 |
| --- | --- | --- | --- |
| Gemma 4 E2B | 4.20GB | 7.2GB | 128K |
| Gemma 4 E4B | 5.90GB | 9.6GB | 128K |
| Gemma 4 26B A4B | 17.00GB | 18GB | 256K |
| Gemma 4 31B | 19.00GB | 20GB | 256K |

읽는 법:

- LM Studio와 Ollama는 같은 모델이라도 배포 포맷과 양자화 방식이 달라 표시 크기가 다를 수 있습니다.
- 일반 사용자가 LM Studio나 Ollama에서 받는 로컬 실행용 모델은 보통 원본 BF16 가중치 그대로가 아니라 4-bit, 5-bit, 8-bit 등으로 양자화된 배포본입니다.
- Ollama의 `gemma4:latest`는 현재 E4B 계열로 표시되지만, `latest`는 바뀔 수 있으므로 명시적 태그를 사용하세요.
- 실행 중 메모리는 모델 크기에 더해 컨텍스트 길이, KV cache, 이미지 입력, 실행 도구 설정에 따라 늘어납니다.

## 모델 이름의 `B`는 무엇인가요?

`B`는 `Billion`, 즉 **10억 개**를 뜻합니다.  
`26B`는 대략 260억 개 파라미터를 가진 모델이고, `31B`는 대략 310억 개 파라미터를 가진 모델입니다.

파라미터는 모델이 학습하면서 갖게 된 숫자 값입니다.  
모델은 이 숫자들을 바탕으로 문장을 이어 쓰고, 질문에 답하고, 코드를 만듭니다.

헷갈리기 쉬운 점:

- `B`는 모델 내부 숫자 규모입니다.
- `GB`는 파일 크기나 메모리 용량 단위입니다.
- `26B`라고 해서 다운로드 파일이 26GB라는 뜻은 아닙니다.
- 실제 표시 크기는 양자화 방식과 배포 포맷에 따라 달라집니다.

그래서 모델을 고를 때는 이름의 `B`만 보지 말고, 위 표의 로컬 배포 크기와 실제 노트북 메모리를 같이 봐야 합니다.

## `26B A4B`에서 `A4B`는 무엇인가요?

`A4B`는 **Active 4B** 로 이해하면 됩니다. 모델 전체는 26B 규모이지만, 답을 만들 때 실제로 계산에 참여하는 부분은 약 4B 규모라는 의미입니다. 공식 모델 카드 기준 수치는 `3.8B`입니다.

이런 구조를 `MoE(Mixture-of-Experts)`라고 부릅니다. 모델 안에 여러 "전문가" 부분이 있고, 답을 만들 때마다 필요한 일부 전문가만 골라 계산합니다.

중요한 점은 `A4B`가 **메모리 사용량이 4B 모델만큼 작다**는 뜻이 아니라는 것입니다. `26B A4B`는 계산할 때 일부 전문가만 쓰지만, 빠른 추론을 위해 전체 26B 규모의 모델 파일을 메모리에 올려야 합니다. 그래서 로컬 배포 크기도 E4B보다 훨씬 큽니다.

## 메모리 계획 포인트

- E2B, E4B는 이름에 보이는 크기보다 모델 파일이 더 클 수 있습니다. 작은 모델을 효율적으로 쓰기 위한 추가 구조가 포함되어 있기 때문입니다.
- 26B A4B는 계산할 때 일부만 쓰는 구조이지만, **전체 모델 파일을 메모리에 올려야** 하므로 메모리 요구량이 4B 모델처럼 작아지지 않습니다.
- 긴 컨텍스트는 이전 대화 내용을 빠르게 이어가기 위한 내부 저장 공간 때문에 메모리를 더 사용합니다.
- fine-tuning은 inference보다 훨씬 더 많은 메모리가 필요합니다.

## 모델 추론에 필요한 메모리는 어떻게 계산하나요?

모델 추론에 필요한 메모리는 크게 아래 항목을 더해서 생각하면 됩니다.

```text
추론 메모리 ≈ 모델 가중치 + KV cache + 입력/출력 처리 메모리 + 실행 도구 오버헤드
```

이 중에서 가장 큰 부분은 보통 **모델 가중치**입니다.  
가중치는 모델이 학습한 파라미터 숫자들을 말합니다.

가중치 메모리는 아래처럼 대략 계산합니다.

```text
가중치 메모리 ≈ 파라미터 수 x 파라미터 1개당 저장 크기
```

자주 보는 숫자 형식:

| 형식 | 파라미터 1개당 크기 | 26B 모델 가중치만 단순 계산 |
| --- | --- | --- |
| FP32 | 4 bytes | 약 104GB |
| BF16 / FP16 | 2 bytes | 약 52GB |
| 8-bit | 1 byte | 약 26GB |
| 4-bit | 0.5 byte | 약 13GB |

즉 26B 모델을 BF16으로 저장하면 가중치만 약 52GB가 필요하지만, 4-bit 양자화 모델은 가중치 기준으로 약 13GB까지 줄어들 수 있습니다.

```text
26B BF16 ≈ 52GB
26B 4-bit ≈ 13GB
```

여기서 52GB가 13GB가 되는 이유는 2 bytes였던 숫자를 0.5 byte 수준으로 줄이기 때문입니다.  
즉 `파라미터 수를 줄이는 것`이 아니라, **파라미터 하나를 저장하는 크기를 줄이는 것**입니다.

다만 실제 추론 메모리는 13GB에서 끝나지 않습니다.  
양자화 메타데이터, KV cache, 컨텍스트 길이, 이미지 입력, 실행 도구 오버헤드가 추가됩니다. 그래서 26B 계열 로컬 배포 크기가 17~18GB처럼 보일 수 있고, 실제 실행 중에는 더 많은 여유 메모리가 필요할 수 있습니다.

NVFP4처럼 scale 정보를 함께 쓰는 4-bit floating point 데이터 타입은 실제로 파라미터 1개당 정확히 4-bit만 쓰는 것은 아닙니다. NVIDIA 설명 기준 NVFP4는 4-bit 값에 scale overhead가 붙어 유효 bit 폭이 약 4.5 bits/value에 가깝습니다. 그래도 BF16/FP16보다 훨씬 작기 때문에 큰 모델을 더 작은 GPU 메모리에 올릴 수 있습니다.

다만 "31B NVFP4가 24GB GPU에서 된다"는 말은 짧은 context와 특정 런타임 조건이 붙은 고급 실험으로 봐야 합니다. 긴 256K context, 이미지/비디오 입력, 큰 batch까지 쓰면 KV cache와 activation 메모리가 커져서 24GB를 넘길 수 있습니다.

## 준비용 간단 계산식

아래 계산식은 **공식 수치가 아니라 준비용 간단 추정식**입니다.

### 1. 사용 가능 메모리 계산

```text
사용 가능 메모리 = 전체 메모리 - 시스템/앱 여유분 - 컨텍스트 여유분
```

권장 기본값:

- 시스템/앱 여유분: `4~6GB`
- 컨텍스트 여유분: `1~3GB`

여유 있게 잡은 권장값:

```text
사용 가능 메모리 ≈ 전체 메모리 - 6GB - 2GB
```

즉,

```text
사용 가능 메모리 ≈ 전체 메모리 - 8GB
```

이 값보다 **모델의 대략적 로드 메모리**가 작아야, 현장에서 비교적 안정적으로 쓸 가능성이 높습니다.

## 가장 쉬운 선택 공식

LM Studio를 기준으로 준비할 때는 아래처럼 생각하면 됩니다.

```text
권장 전체 메모리 ≈ 로컬 모델 표시 크기 + 8GB
```

예시:

- E2B: `4.2GB + 8GB ≈ 12.2GB`
- E4B: `5.9GB + 8GB ≈ 13.9GB`
- 26B A4B: `17GB + 8GB ≈ 25GB`
- 31B: `19GB + 8GB ≈ 27GB`

Ollama 기본 태그는 같은 모델이라도 표시 크기가 더 클 수 있습니다. 특히 16GB 장비에서 `gemma4:e4b`가 버겁게 느껴지면 `gemma4:e2b`로 낮추세요.

이 계산은 "행사에서 너무 빡빡하지 않게 쓰는 기준"입니다.  
즉:

- E2B는 8GB에서는 **거의 최소치이며 속도 저하 가능**
- E4B는 16GB에서 현실적
- 26B A4B, 31B는 32GB 이상에서 현실적

## RAM만 보면 되나요?

항상 그렇지는 않습니다.

- **Apple Silicon Mac**: unified memory 구조라서, 보통은 이 문서의 "전체 메모리 기준 계산"을 그대로 적용해도 됩니다.
- **Windows/Linux + 전용 GPU**: GPU 가속을 쓸 때는 시스템 RAM과 별개로 VRAM이 부족할 수 있습니다.
- 예를 들어 시스템 메모리가 32GB여도 VRAM이 작으면 큰 모델은 GPU에 충분히 못 올라가서 속도가 느려지거나, 일부 오프로딩으로 체감이 달라질 수 있습니다.
- 전용 GPU 사양이 애매하면 **더 작은 모델(E2B/E4B)** 을 고르세요.

## 메모리별로 어떻게 고르면 되나요?

### 8GB

간단 계산:

```text
사용 가능 메모리 ≈ 8GB - 8GB = 0GB
```

실제로는 아주 빡빡합니다.

해석:

- E2B도 운영체제와 앱이 같이 뜬 상태에서는 여유가 거의 없습니다.
- 따라서 8GB는 **E2B만 시도하고, 속도 저하를 감안**해 주세요.
- Windows/macOS에서 LM Studio 공식 권장은 16GB 이상이라, 8GB는 "무난한 기본 사양"이 아닙니다.
- 특히 8GB Apple Silicon Mac이나 8GB 일반 노트북에서는 E2B/E4B 실행 시 전체 시스템이 매우 느려지거나 일시적으로 멈춘 것처럼 보일 수 있습니다.

권장:

- `E2B`만 준비
- 가능하면 16GB 이상 장비 사용
- 브라우저 탭 최소화
- 다른 앱 종료
- 가능하면 전원 연결

### 16GB

간단 계산:

```text
사용 가능 메모리 ≈ 16GB - 8GB = 8GB
```

해석:

- `E4B`는 LM Studio로 준비할 때 현실적인 선택
- `E2B`는 더 여유 있음
- `26B A4B`, `31B`는 사실상 어렵습니다

권장:

- 기본: `E4B`
- 실패 시: `E2B`

### 32GB

간단 계산:

```text
사용 가능 메모리 ≈ 32GB - 8GB = 24GB
```

해석:

- `E4B`는 매우 여유 있음
- `26B A4B`는 현실적
- `31B`도 가능 범위에 들어오지만, 긴 컨텍스트와 다른 앱까지 생각하면 여유가 줄어듭니다

권장:

- 안정성 우선: `E4B`
- 성능 우선: `26B A4B`
- 실험적 상위 옵션: `31B`

### 36GB

간단 계산:

```text
사용 가능 메모리 ≈ 36GB - 8GB = 28GB
```

해석:

- `26B A4B`, `31B` 모두 현실적
- 31B도 가능성이 높지만, 속도와 반응성은 26B A4B가 더 나은 경우가 있을 수 있습니다

권장:

- 균형형: `26B A4B`
- 품질 우선: `31B`

## GGUF / MLX / 도구별로 이 표를 어떻게 보면 되나요?

- **Ollama / LM Studio GGUF**: 보통 4비트~8비트 계열 로컬 모델을 많이 쓰지만, 도구별 표시 크기와 실제 메모리 사용량은 다를 수 있습니다.
- **MLX**: Apple Silicon에서 잘 맞는 선택지이지만, 실제 메모리 사용량은 모델 변환 방식과 정밀도에 따라 달라질 수 있습니다.
- 즉, 위 표는 **준비용 기준점**이고, 실제 파일 크기와 런타임 메모리는 도구별로 조금씩 다릅니다.

## 한 줄 정리

- 8GB: `E2B만 시도, 속도 저하 감안`
- 16GB: `E4B`
- 32GB: `E4B 또는 26B A4B`
- 36GB: `26B A4B 또는 31B`
