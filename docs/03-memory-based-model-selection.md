# 메모리 기준으로 Gemma 4 모델 고르는 방법

[메인 안내로 돌아가기](../gemma4-local-setup-guide.md)

## 이 문서에서 다루는 내용

- 공식 Gemma 4 메모리 요구사항 표
- 노트북/데스크톱 메모리로 어떤 모델을 고르면 되는지
- 행사 준비용 계산식
- 8GB, 16GB, 32GB, 36GB 환경별 예시

## 먼저 알아둘 점

- 아래 공식 표는 **모델 가중치(weights)를 메모리에 올리는 데 필요한 대략적인 GPU/TPU 메모리** 기준입니다.
- 공식 문서도 이 수치가 **도구와 환경에 따라 달라질 수 있다**고 설명합니다.
- 또한 이 수치는 **Base weights only** 기준이라서, 운영체제, 런타임, 브라우저, KV cache, 긴 컨텍스트, 멀티모달 입력으로 인한 추가 메모리는 포함하지 않습니다.
- 따라서 실제 참가자용 노트북에서는 **표의 숫자보다 여유를 더 잡아야** 합니다.
- Apple Silicon Mac처럼 **unified memory**를 쓰는 환경에서는 전체 메모리를 하나의 풀처럼 보면 됩니다.
- Windows/Linux의 **전용 GPU 노트북**은 시스템 메모리(RAM)뿐 아니라 **GPU 메모리(VRAM)** 도 같이 확인하는 편이 안전합니다.

## 공식 Gemma 4 Inference Memory Requirements

Gemma 4 모델 개요 문서 기준:

| 모델 | BF16 (16-bit) | SFP8 (8-bit) | Q4_0 (4-bit) |
| --- | --- | --- | --- |
| Gemma 4 E2B | 9.6 GB | 4.6 GB | 3.2 GB |
| Gemma 4 E4B | 15 GB | 7.5 GB | 5 GB |
| Gemma 4 31B | 58.3 GB | 30.4 GB | 17.4 GB |
| Gemma 4 26B A4B | 48 GB | 25 GB | 15.6 GB |

읽는 법:

- BF16은 기본 정밀도라 로컬 노트북에서는 대체로 무겁습니다.
- SFP8은 8비트 계열입니다.
- Q4_0은 4비트 계열 예시입니다.
- 실제 LM Studio, Ollama, GGUF, MLX 환경에서는 **정확한 파일 크기와 메모리 사용량이 조금씩 다를 수 있습니다.**

## 공식 문서가 강조하는 메모리 계획 포인트

- E2B, E4B는 "effective parameter" 구조라, 체감 파라미터 수보다 정적 가중치 메모리가 더 커질 수 있습니다.
- 26B A4B는 MoE이지만, 추론 시 빠른 라우팅을 위해 **전체 파라미터를 메모리에 올려야** 하므로 메모리 요구량이 4B 모델처럼 작아지지 않습니다.
- 긴 컨텍스트는 KV cache 때문에 메모리를 더 사용합니다.
- fine-tuning은 inference보다 훨씬 더 많은 메모리가 필요합니다.

## 행사 준비용 실무 계산식

아래 계산식은 **공식 수치가 아니라 행사 운영용 휴리스틱**입니다.

### 1. 사용 가능 메모리 계산

```text
사용 가능 메모리 = 전체 메모리 - 시스템/앱 여유분 - 컨텍스트 여유분
```

권장 기본값:

- 시스템/앱 여유분: `4~6GB`
- 컨텍스트 여유분: `1~3GB`

행사 기준 보수적 권장값:

```text
사용 가능 메모리 ≈ 전체 메모리 - 6GB - 2GB
```

즉,

```text
사용 가능 메모리 ≈ 전체 메모리 - 8GB
```

이 값보다 **모델의 대략적 로드 메모리**가 작아야, 현장에서 비교적 안정적으로 쓸 가능성이 높습니다.

## 가장 쉬운 선택 공식

4비트 계열 로컬 모델을 쓴다고 가정하면, 행사 준비용으로는 아래처럼 생각하면 됩니다.

```text
권장 전체 메모리 ≈ 모델(Q4_0급) 메모리 + 8GB
```

예시:

- E2B: `3.2GB + 8GB ≈ 11.2GB`
- E4B: `5GB + 8GB ≈ 13GB`
- 26B A4B: `15.6GB + 8GB ≈ 23.6GB`
- 31B: `17.4GB + 8GB ≈ 25.4GB`

이 계산은 "행사에서 너무 빡빡하지 않게 쓰는 기준"입니다.  
즉:

- E2B는 8GB에서는 **거의 최소치 / best-effort**
- E4B는 16GB에서 현실적
- 26B A4B, 31B는 32GB 이상에서 현실적

## RAM만 보면 되나요?

항상 그렇지는 않습니다.

- **Apple Silicon Mac**: unified memory 구조라서, 보통은 이 문서의 "전체 메모리 기준 계산"을 그대로 적용해도 됩니다.
- **Windows/Linux + 전용 GPU**: GPU 가속을 쓸 때는 시스템 RAM과 별개로 VRAM이 부족할 수 있습니다.
- 예를 들어 시스템 메모리가 32GB여도 VRAM이 작으면 큰 모델은 GPU에 충분히 못 올라가서 속도가 느려지거나, 일부 오프로딩으로 체감이 달라질 수 있습니다.
- 참가자 안내 기준으로는, 전용 GPU 사양이 애매하면 **더 작은 모델(E2B/E4B)** 을 고르는 편이 안전합니다.

## 메모리별로 어떻게 고르면 되나요?

### 8GB

간단 계산:

```text
사용 가능 메모리 ≈ 8GB - 8GB = 0GB
```

실제로는 아주 빡빡합니다.

해석:

- 공식 표의 `E2B Q4_0 = 3.2GB`만 봐도, 운영체제와 앱이 같이 뜬 상태에서는 여유가 거의 없습니다.
- 따라서 8GB는 **E2B만 best-effort** 로 보는 것이 안전합니다.
- Windows/macOS에서 LM Studio 공식 권장은 16GB 이상이라, 8GB는 "무난한 기본 사양"이 아닙니다.
- 특히 8GB Apple Silicon Mac이나 8GB 일반 노트북에서는 E2B/E4B 실행 시 전체 시스템이 매우 느려지거나 일시적으로 멈춘 것처럼 보일 수 있습니다.

권장:

- `E2B`만 준비
- 가능하면 16GB 이상 장비 사용
- 브라우저 탭 최소화
- 다른 앱 종료
- 가능하면 전원 연결

### 16GB

간단 계산:

```text
사용 가능 메모리 ≈ 16GB - 8GB = 8GB
```

해석:

- `E4B Q4_0 = 5GB`는 현실적인 선택
- `E2B`는 더 여유 있음
- `26B A4B Q4_0 = 15.6GB`, `31B Q4_0 = 17.4GB`는 사실상 어렵습니다

권장:

- 기본: `E4B`
- 실패 시: `E2B`

### 32GB

간단 계산:

```text
사용 가능 메모리 ≈ 32GB - 8GB = 24GB
```

해석:

- `E4B`는 매우 여유 있음
- `26B A4B Q4_0 = 15.6GB`는 현실적
- `31B Q4_0 = 17.4GB`도 가능 범위에 들어오지만, 긴 컨텍스트와 다른 앱까지 생각하면 여유가 줄어듭니다

권장:

- 안정성 우선: `E4B`
- 성능 우선: `26B A4B`
- 실험적 상위 옵션: `31B`

### 36GB

간단 계산:

```text
사용 가능 메모리 ≈ 36GB - 8GB = 28GB
```

해석:

- `26B A4B`, `31B` 모두 현실적
- 31B도 가능성이 높지만, 속도와 반응성은 26B A4B가 더 나은 경우가 있을 수 있습니다

권장:

- 균형형: `26B A4B`
- 품질 우선: `31B`

## GGUF / MLX / 도구별로 이 표를 어떻게 보면 되나요?

- **Ollama / LM Studio GGUF**: 보통 4비트~8비트 계열 로컬 모델을 많이 쓰므로, 공식 표에서는 `Q4_0` 또는 `SFP8` 칸을 **대략적인 출발점**으로 보면 됩니다.
- **MLX**: Apple Silicon에서 잘 맞는 선택지이지만, 실제 메모리 사용량은 모델 변환 방식과 정밀도에 따라 달라질 수 있습니다.
- 즉, 공식 표는 **대략적인 기준점**이고, 실제 파일 크기와 런타임 메모리는 도구별로 조금씩 다릅니다.

## 행사 운영 기준 한 줄 정리

- 8GB: `E2B only, best-effort`
- 16GB: `E4B`
- 32GB: `E4B 또는 26B A4B`
- 36GB: `26B A4B 또는 31B`
