# Gemma 4 벤치마크와 코딩 에이전트 기대치

[메인 안내로 돌아가기](../gemma4-local-setup-guide.md)

## 왜 이 문서가 필요한가요?

Gemma 4는 로컬에서 실행 가능한 매우 강력한 오픈 모델이지만, **모델 크기에 따라 체감 성능 차이**가 큽니다.  
특히 `E2B`, `E4B`는 작은 노트북에서도 돌아가도록 설계된 모델이라, **Codex, Claude Code 같은 상용 코딩 에이전트 수준의 성능을 그대로 기대하면 차이가 크게 느껴질 수 있습니다.**

이번 문서는 공식 Gemma 4 모델 카드 기준 벤치마크를 정리하고, **실습 참가자가 어떤 기대치를 가져야 하는지** 설명하기 위한 문서입니다.

## 공식 벤치마크 요약

아래 수치는 Google 공식 Gemma 4 모델 카드에 나온 수치입니다.

| 모델 | MMLU Pro | LiveCodeBench v6 | Codeforces ELO | Tau2 | MRCR v2 8 needle 128k |
| --- | --- | --- | --- | --- | --- |
| Gemma 4 E2B | 60.0 | 44.0 | 633 | 24.5 | 19.1 |
| Gemma 4 E4B | 69.4 | 52.0 | 940 | 42.2 | 25.4 |
| Gemma 4 26B A4B | 82.6 | 77.1 | 1718 | 68.2 | 44.1 |
| Gemma 4 31B | 85.2 | 80.0 | 2150 | 76.9 | 66.4 |

읽는 법:

- `LiveCodeBench v6`, `Codeforces ELO`는 코딩 성능을 볼 때 참고하기 좋습니다.
- `Tau2`는 에이전트형 작업을 볼 때 참고하기 좋은 지표입니다.
- `MRCR v2 8 needle 128k`는 긴 문맥 처리 능력을 볼 때 참고할 수 있습니다.

## 모델별 해석

### E2B

- 가장 작은 경량 모델입니다.
- 로컬 데모, 간단한 질의응답, 가벼운 보조 작업에는 적합합니다.
- 하지만 코딩과 에이전트 성능은 상위 모델과 차이가 큽니다.
- `LiveCodeBench 44.0`, `Tau2 24.5` 수준이라, **복잡한 코드 수정, 다단계 계획, 도구 사용이 많은 작업에서는 한계가 빨리 드러날 수 있습니다.**

권장 사용:

- 8GB 환경
- 최소 사양 테스트
- “로컬 LLM이 돌아간다”는 확인용

### E4B

- 실습 참가자용 기본 모델로 가장 현실적인 선택지입니다.
- E2B보다 확실히 낫지만, 여전히 **소형 모델**입니다.
- `LiveCodeBench 52.0`, `Tau2 42.2`로 E2B보다 개선되지만, 26B/31B와는 차이가 큽니다.

권장 사용:

- 16GB 전후 노트북
- 일반적인 질의응답, 문서 정리, 간단한 코드 보조
- 행사 표준 기본 모델

주의:

- Codex, Claude Code 같은 강한 코딩 에이전트를 써봤던 참가자라면 **생각보다 답이 약하거나, 툴 사용이 덜 안정적으로 느껴질 수 있습니다.**

### 26B A4B

- 성능과 속도의 균형이 좋은 상위 옵션입니다.
- 공식 모델 카드도 이 모델을 **31B 대비 빠른 로컬 추론에 유리한 선택지**로 설명합니다.
- `LiveCodeBench 77.1`, `Tau2 68.2`로 E4B보다 큰 폭으로 올라갑니다.

권장 사용:

- 32GB 이상
- 코딩, 계획, 에이전트형 흐름을 좀 더 안정적으로 쓰고 싶은 경우
- 31B보다 속도를 더 중시할 때

### 31B

- Gemma 4 계열 중 가장 높은 일반 성능을 보여주는 상위 모델입니다.
- `LiveCodeBench 80.0`, `Codeforces 2150`, `Tau2 76.9`로 공식 표에서 가장 강합니다.
- 하드웨어가 허용하면 가장 좋은 선택입니다.

권장 사용:

- 36GB 이상
- 긴 문맥, 코딩, 복잡한 작업, 더 높은 응답 품질이 중요할 때

## 코딩 에이전트 사용자 관점에서의 안내

공식 Gemma 4 모델 카드는 Gemma 4가 **native function-calling support** 를 갖고 있고, reasoning, coding, agentic workflow에 적합하다고 설명합니다.  
하지만 **모델 크기별 벤치마크 차이**를 보면, 특히 `E2B`, `E4B`는 상위 모델과 성능 차이가 큽니다.

실무적으로는 아래처럼 **공식 수치**와 **실무적 해석**을 나눠서 보는 것이 안전합니다.

공식 수치로 직접 말할 수 있는 부분:

- `E2B`, `E4B`는 `LiveCodeBench`, `Codeforces`, `Tau2`, `MRCR` 같은 코딩·에이전트 프록시 지표에서 26B/31B보다 낮습니다.
- 따라서 작은 모델일수록 복잡한 코드 수정, 긴 지시 처리, 다단계 계획에서 한계가 더 빨리 드러날 가능성이 높습니다.

실무적 해석으로만 말해야 하는 부분:

- Gemma 4는 로컬 실습과 개인 환경에서 매우 유용합니다.
- 하지만 `E2B`, `E4B`에 대해 **Codex, Claude Code 수준의 코딩 에이전트 경험을 그대로 기대하면 안 됩니다.**
- 툴 사용 흐름의 안정성은 공식 벤치마크가 직접 측정한 항목이 아니라, 런타임, 프롬프트 구성, 로컬 백엔드(OpenAI-compatible 서버 포함)에 따라 달라질 수 있습니다.
- 따라서 작은 모델에서는:
  - 코드 수정 정확도가 낮아질 수 있음
  - 긴 작업 지시를 놓칠 수 있음
  - 여러 단계 계획이 흔들릴 수 있음
  - 툴 사용이 필요한 흐름에서 일관성이 떨어진다고 **체감될 수 있음**

주의:

- 위 항목 중 특히 툴 사용 흐름에 대한 평가는 공식 벤치마크가 직접 측정한 값이 아니라, 코딩/에이전트 프록시 지표와 실사용 체감을 바탕으로 한 해석입니다.

## 참가자 안내 문구로 바로 써도 되는 표현

아래 문구는 행사 안내에 그대로 넣어도 무난합니다.

```text
Gemma 4 E2B, E4B는 로컬 환경에서 실행하기 좋은 경량 모델이지만, Codex나 Claude Code 같은 상용 코딩 에이전트 수준의 성능을 그대로 기대하면 차이가 크게 느껴질 수 있습니다.

특히 E2B, E4B는 코드 생성/수정과 다단계 계획에서 상위 모델(26B, 31B)보다 성능 차이가 날 수 있습니다. 툴 사용 흐름은 공식 벤치마크의 직접 측정 항목은 아니며, 실제 안정성은 런타임과 프롬프트 구성에 따라 달라질 수 있습니다. 이번 핸즈온에서는 “로컬에서 실행 가능한 Gemma 4 체험”과 “기본 워크플로우 이해”에 초점을 맞춰 주세요.
```

## 행사 운영 관점의 권장 해석

- 8GB: E2B는 “최소 실행 보장”에 가깝습니다.
- 16GB: E4B는 가장 현실적인 기본 모델입니다.
- 32GB 이상: 코딩/에이전트 체감을 더 원하면 26B A4B를 권장합니다.
- 36GB 이상: 가능하면 31B가 가장 좋습니다.
- 모델별 구조 차이를 더 자세히 보려면 [Gemma 4 아키텍처 상세 정리](./14-gemma4-architecture-deep-dive.md) 문서를 확인해 주세요.

## OpenCode 같은 코딩 에이전트에 연결할 때

- `opencode` 자체는 로컬 모델과 LM Studio를 공식적으로 지원하므로, 개발자 에이전트 시연 용도로는 **조건부로 적절합니다.**
- 다만 OpenCode 공식 문서도 코드 생성과 tool calling을 모두 잘하는 모델은 많지 않다고 설명합니다.
- 따라서 Gemma 4를 `opencode`에 붙일 때는:
  - `E2B`: 비추천
  - `E4B`: 가벼운 보조 수준
  - `26B A4B`, `31B`: 더 현실적인 선택
- 자세한 설치와 연동 방법은 [OpenCode 설치 및 LM Studio 연동 가이드](./10-opencode-lmstudio-developer-agent.md) 문서를 확인해 주세요.
